# Pass 1b — Database Audit
*Phase 1 analysis | 2026-05-22*

---

## Stated Architectural Intents

1. **SQLite-first authority** — all runtime STATE must be SQLite-backed. File-based state is v1 rot.
2. **Security audit during brownfield onboarding** — security skills run during project intake; findings stored in SQLite.
3. **Security audit as SDLC lifecycle gate** — greenfield projects must pass security audit before going live.
4. **Canonical events as the spine** — all state changes flow through `canonical_events`. Direct table writes without event emission are anomalies.
5. **Marker file authority for attribution** — `.dream-studio-project` markers are identity source; `ds_projects` is metadata storage.

## Special Focus: File-Based vs SQLite State

### 1. `workflows.json` — Workflow Active State

- **Path:** `~/.dream-studio/state/workflows.json`
- **Content:** `{"schema_version": 1, "active_workflows": {}}`
- **What it stores:** Currently running workflow state (active workflow key, current node, status)
- **What uses it:** `ds workflow status`, `ds workflow advance`, `ds workflow list` — all read this file, not the DB
- **Corresponding SQLite table:** `raw_workflow_runs` (exists, 2 rows) — but it stores completed/historical runs, not live state
- **Gap:** Live workflow state is file-only. There is no SQLite table for "currently executing" workflow state. `raw_workflow_runs` becomes the record after completion; during execution the file is the only authority.
- **Severity:** HIGH — violates Intent #1. A crash during workflow execution leaves `workflows.json` as the only recovery point. The `workflow-checkpoint.json` file (which records `{"workflow_key": "wf-fail-1", "last_node": "n1", "status": "failed"}`) is also file-only.

### 2. `active_task.json` — Active Task Context

- **Path:** `~/.dream-studio/state/active_task.json`
- **What it stores:** The currently active SDLC task with full project/milestone/work_order context chain
- **CLI commands that use it:** `ds task set-active` (writes), `ds task active` (reads), `ds task clear-active` (deletes)
- **Corresponding SQLite table:** `ds_tasks` exists and has 9 rows, but it does not have an `is_active` column or equivalent
- **Gap:** Active task attribution is fully file-based. There is a `task_attribution_records` table with appropriate columns, but it is empty and is not the read path for `ds task active`.
- **Severity:** MEDIUM — Intent #1 violation. The active task pointer is not queryable from the DB. Any tool that needs to know the active task must read the file.

### 3. `hook-timing.jsonl` — Hook Performance Telemetry

- **Path:** `~/.dream-studio/state/hook-timing.jsonl`
- **Size:** 387 KB, 3,596 lines
- **What it stores:** Per-hook execution timing (event, handler, duration_ms, timestamp)
- **Written by:** `on-tool-activity.py` hook
- **Corresponding SQLite table:** `hook_executions` (45 rows — only on_pulse) and `hook_invocations` (917 rows — all on-tool-activity)
- **Gap:** 3,596 timing records exist in the file; only 45 are in hook_executions and those are a different hook. The 917 hook_invocations rows do not contain timing data. The primary hook telemetry data lives in the .jsonl file.
- **Severity:** HIGH — Intent #1 violation. The `.jsonl` file is the real hook performance authority. Dashboard `/api/v1/hooks/performance` reads hook_executions which has stale, incomplete data.

### 4. `skill-usage.jsonl` and `telemetry-buffer.jsonl` — Skill Telemetry

- **Path:** `~/.dream-studio/state/skill-usage.jsonl` (3 lines, two schema formats), `~/.dream-studio/state/telemetry-buffer.jsonl` (1 line)
- **Written by:** `on-skill-telemetry.py` hook (Stop trigger)
- **Corresponding SQLite table:** `raw_skill_telemetry` (exists, 0 rows), `skill_invocations` (1 row — manual session telemetry, not hook-generated)
- **Gap:** The skill telemetry hook writes to `.jsonl` files. The `raw_skill_telemetry` DB table is never written to by the live hook. Additionally, the two `.jsonl` files have different schema formats (inconsistency between invocations).
- **Severity:** MEDIUM — Intent #1 violation, plus schema inconsistency in the file itself.

### 5. `token-capture.jsonl` — Token Capture Diagnostics

- **Path:** `~/.dream-studio/state/diagnostics/token-capture.jsonl`
- **Size:** 3,690 bytes
- **Written by:** `on-post-tool-use.py` core hook via `core.telemetry.token_capture`
- **Corresponding SQLite table:** `raw_token_usage` (3 rows, all zeros), `token_usage_records` (0 rows)
- **Gap:** Token capture data goes to the .jsonl diagnostic file. The DB tables exist but are not populated by the token capture hook.
- **Severity:** MEDIUM — Intent #1 violation.

### 6. `.sessions/` directory — Session Handoffs and Recaps

- **Path:** `~/.dream-studio/.sessions/YYYY-MM-DD/handoff-<hash>.md`, `recap-<hash>.md`
- **What it stores:** Post-session handoff notes and session recaps, written by `on-stop-handoff.py`
- **Corresponding SQLite table:** `raw_handoffs` (26 rows) — this table IS populated
- **Alignment status:** DUAL STORE — both the files and the DB are written. The `.sessions/` files are richer (full markdown) while raw_handoffs presumably stores structured fields. Not a pure file-only store, but the canonical content is in the files.

### 7. `.dream-studio-project` Marker Files — Project Identity

- **What it stores:** Project identity marker at project root (Intent #5 explicitly designates this as the authority)
- **Corresponding SQLite table:** `ds_projects` (25 rows — metadata storage per Intent #5)
- **Alignment status:** INTENDED DUAL STORE — this is the stated architectural model, not a gap. The marker is identity authority; `ds_projects` is metadata. However, `reg_projects` also stores project paths (2 rows with project_path field), creating a third source of project identity information.

## Special Focus: Dual Storage Stores

### reg_projects vs ds_projects

These tables store fundamentally different things:

| Aspect | reg_projects | ds_projects |
|--------|-------------|-------------|
| Primary key | project_id (TEXT — path-based: "dream-studio-clean", "Dannis Seay") | project_id (UUID) |
| Contents | File path, health scores, stack analysis, session counts, security scores, adapter info | Name, status, description, timestamps |
| Row count | 2 | 25 |
| Written by | `spool.session_harvester.SessionHarvester` (during memory ingest) | `core.projects.mutations.register_project()` (CLI: project register) |
| Purpose | Runtime intelligence about a directory on disk | SDLC project entity (work orders, milestones hang off this) |

**Assessment:** These are NOT duplicates. They are orthogonal: reg_projects is a path-indexed operational intel record (like a "what's installed here"); ds_projects is a UUID-indexed SDLC entity. The confusion risk is the name similarity. `reg_projects.project_id = "dream-studio-clean"` (a slug) vs `ds_projects.project_id = "a4befdce-..."` (a UUID) make them incompatible as foreign keys to each other.

**Finding:** There is no formal join between reg_projects and ds_projects. A project registered via `ds project register` (writes to ds_projects) is a different entity than what the session harvester writes to reg_projects. The telemetry and SDLC systems operate on different project identity models with no reconciliation layer.

### raw_workflow_runs vs workflow_invocations

Both tables have 2 rows, both representing the same two studio-onboard runs.

| Aspect | raw_workflow_runs | workflow_invocations |
|--------|------------------|---------------------|
| Schema origin | Early workflow tracking | Execution spine migration (037) |
| Primary key | id (auto-int) | invocation_id (TEXT UUID) |
| Status values | aborted, completed | cancelled, completed |
| Metadata | Flat columns: status, started_at, finished_at, node_count | metadata_json blob containing entire node execution tree |
| Evidence refs | None | evidence_refs_json pointing to raw_workflow_runs |

**Assessment:** These ARE duplicates, but with different purposes. `raw_workflow_runs` is the lightweight tracking table written by the workflow runner during execution. `workflow_invocations` is the canonical execution spine record written post-completion, which references raw_workflow_runs in its evidence_refs_json. The architecture intends for `workflow_invocations` to be the authority (execution spine) and `raw_workflow_runs` to be raw operational data. However, both store the same run; the raw table is not deleted after the invocation record is created.

**Finding:** workflow_invocations.metadata_json contains the full node execution tree (hundreds of characters per row), making it a large blob store rather than a relational structure. This data would be better served by the empty `execution_nodes` and `execution_outputs` tables, which exist but have never been written to.

### skill_invocations vs reg_skills

- `skill_invocations`: 1 row (session telemetry record, skill=unknown). Written by the session telemetry hook path.
- `reg_skills`: 0 rows. Intended to be populated by `hydrate_registry.py` with the skill catalog.

**Assessment:** These are NOT the same thing. reg_skills is a static catalog ("what skills exist"), skill_invocations is a runtime record ("what skills were called"). They are complementary, not duplicate. Neither is functioning — reg_skills has no data and skill_invocations has only one meaningless record with skill=unknown.

### hook_invocations vs tool_invocations

Both have exactly 917 rows. Both are populated by the `on-tool-activity` hook. Column inspection shows:

- `hook_invocations`: invocation_id, project_id, milestone_id, task_id, process_run_id, event_id, hook_id, status, prevented_risky_action, purpose, metadata_json
- `tool_invocations`: invocation_id, project_id, milestone_id, task_id, process_run_id, event_id, tool_id, status, purpose, metadata_json

**Assessment:** These ARE structural duplicates with overlapping data. The on-tool-activity hook appears to write one record to each table per tool use. The hook_invocations record has `hook_id='on-tool-activity'`; the tool_invocations record has `tool_id=<tool_name>` (e.g., PowerShell, Bash, Read). They share the same event_id. This is intentional modeling (hook view vs tool view of the same event) but produces a doubling of rows for every tool use with no join table.

| `research_cache` | INSERT, DELETE | `POST /api/discovery/research`, `DELETE /api/discovery/research/{id}` | LOW — cache is ephemeral |
| `alert_rules` | INSERT, UPDATE, DELETE | `POST/PUT/DELETE /api/v1/alerts/rules` | MEDIUM — alert rule changes invisible to event spine |
| `audit_runs` | INSERT | `POST /api/v1/audits/runs` | MEDIUM — audit lifecycle not event-traced |
| `sec_sarif_findings` | INSERT (TODO/stubbed) | `POST /api/v1/security/sarif/import` | HIGH — security findings without events breaks Intent #3 |

### Commands Emitting Events Correctly

For reference, these commands do emit canonical events via spool:
- `project register` → `project.created`
- `project delete` → `project.deleted`, `task.deleted` per task
- `project start` → `work_order.started`
- `work-order start` → `work_order.started`
- `work-order close` → `work_order.closed`, optionally `gate.bypassed`
- `work-order block` → `work_order.blocked`
- `work-order task-done` → `task.completed`
- `work-order add-tasks` → `task.created` per task
- `milestone close` → `milestone.completed`, optionally `gate.bypassed`
- `skill invoke` → `skill.invoked` (best-effort)

**Finding:** Design brief lifecycle is event-silent. A design brief being created, updated, locked, or unlocked generates no canonical event. This is notable because the design brief gate (`design_brief_locked`) is a hard precondition for UI work orders — but the moment of locking is invisible to the event spine. If a brief is re-opened or modified, there is no audit trail.

**Finding:** project set-active and project deactivate emit no events. These are state transitions on the primary SDLC entity. The active project changes silently. Given that "active project" determines what all subsequent commands operate on, this is a high-confidence gap.

## Special Focus: Security Tables State

All security tables are empty. This is a categorical violation of Intents #2 and #3.

### Root Cause Analysis

**sec_sarif_findings:** The only write path is `POST /api/v1/security/sarif/import`, which the Phase 0c inventory explicitly notes is STUBBED: "parse_sarif_file() stub not yet wired." The API endpoint accepts file uploads but does not complete the insert. Even if the stub were wired, the API would only be called by external tooling that pushes SARIF output.

**sec_cve_matches:** No write path identified in CLI or API mappings. This table would be populated by a dependency scanner (pip-audit, npm audit, etc.) integrated into the hook or workflow chain. No such integration exists.

**sec_hook_checks:** Schema references `activity_id` and `hook_exec_id`. These columns suggest this table was meant to be populated by the security check hook (`on-security-scan.py`). However, `on-security-scan.py` is documented as "Advisory only" in the hook inventory — it nudges on patterns but does not write to the DB.

**sec_manual_reviews:** No write path identified. Would require a formal review workflow to produce a structured record.

**security_findings:** No write path identified in live hooks or CLI. The columns include `introduced_by_agent_id`, `introduced_by_skill_id`, `introduced_by_workflow_id`, `introduced_by_hook_id` — this table was designed for fine-grained attribution of where a finding was introduced, but nothing writes to it.

**guardrail_decisions / guardrail_rules_audit:** The `guardrails/` module (enforcement.py, evaluator.py, loader.py) exists and has rules defined in YAML. The guardrail scanners are all stubs (giskard, llm_guard, rebuff require Python <3.12). The guardrail system has never written a decision record.

### Downstream Impact

- `GET /api/v1/security/findings` → returns empty
- `GET /api/v1/security/sarif` → returns empty
- `GET /api/v1/security/cve` → returns empty
- `GET /api/v1/security/reviews` → returns empty
- `GET /api/v1/security/stats` → returns 0 for all counts
- `GET /api/v1/projects/{id}/security` → returns empty

The entire security dashboard surface is non-functional not due to query bugs but because the write paths that should populate these tables do not exist in the current system. The `studio-onboard` workflow's completed run explicitly noted "8 assets to adopt" and "PERSISTENCE_GAP, TRACKING_GAP workflows" — the gap was identified at onboarding time and remains open.

## Findings

1. **F01 — Test data in production DB.** 23 of 25 ds_projects rows are test fixtures from pytest runs. Tests are writing to the live `~/.dream-studio/state/studio.db`. Test isolation is not enforced at the DB level. All 23 fixture rows have status='paused' (from a backfill migration) but they pollute project listings and count-based queries.

2. **F02 — Active project mismatch between MEMORY.md and DB.** MEMORY.md states Dream Command is the active build project, but `ds_projects` shows "Dream Studio" as status='active' and "Dream Command" as status='paused'. The `project set-active` command that would fix this emits no event (F10 below).

3. **F03 — Live workflow state is file-only.** `workflows.json` and `workflow-checkpoint.json` are the only authority for currently executing or last-failed workflow state. No SQLite table captures live workflow execution state. Violates Intent #1.

4. **F04 — Active task pointer is file-only.** `active_task.json` is the sole authority for the currently active SDLC task. The `ds_tasks` table has no `is_active` column. `task_attribution_records` exists but is empty and is not in the read path for `ds task active`. Violates Intent #1.

5. **F05 — Hook timing telemetry is file-only.** 3,596 hook timing records exist in `hook-timing.jsonl` (387 KB). `hook_executions` has only 45 rows (on_pulse only). The hook performance dashboard reads `hook_executions`, which is severely under-populated relative to the actual hook execution history. Violates Intent #1.

6. **F06 — Skill telemetry is file-only.** `skill-usage.jsonl` and `telemetry-buffer.jsonl` contain the skill invocation record; `raw_skill_telemetry` is empty. The on-skill-telemetry hook writes files, not the DB. Violates Intent #1.

7. **F07 — Token capture is file-only.** `token-capture.jsonl` captures token usage; `raw_token_usage` has only 3 rows with all-zero values. `token_usage_records` is empty. Violates Intent #1.

8. **F08 — hook_invocations/tool_invocations bypass the event spine.** 917 rows in each table were written by `on-tool-activity`, but only 246 `hook.tool_activity` events appear in canonical_events. The write path for these tables does not go through spool → ingest. These tables are direct DB writes without canonical event emission. Violates Intent #4.

9. **F09 — validation_failures reveal a persistent hook schema violation.** 57 events were rejected because `execution.completed` does not match the required `domain.entity.action` three-part format. Source is the pulse hook. The violation was active from 2026-05-17 to 2026-05-19; the hook appears to have stopped running after 2026-05-19 (hook_executions last entry is 2026-05-19).

10. **F10 — Design brief lifecycle is event-silent.** `design-brief create`, `design-brief lock`, `design-brief update`, and `design-brief set-system` all write to `ds_design_briefs` without emitting canonical events. The brief locking gate is a critical SDLC control point; the absence of events means brief state transitions leave no audit trail. Violates Intent #4.

11. **F11 — Project state transitions are event-silent.** `project set-active` and `project deactivate` modify `ds_projects.status` without emitting events. Violates Intent #4.

12. **F12 — Security tables are entirely empty.** All sec_* tables (cve_matches, hook_checks, manual_reviews, sarif_findings), security_findings, and guardrail tables have 0 rows. The SARIF import endpoint is stubbed. The CVE and manual review write paths do not exist in the current system. Violates Intents #2 and #3 categorically.

13. **F13 — reg_skills and reg_workflows are empty.** The skill and workflow registry tables have never been hydrated. `hydrate_registry.py` exists but has not been run against the live DB. The skill list CLI reads the filesystem directly, making the registry tables orphaned infrastructure.

14. **F14 — PRD system has never been used.** All prd_* tables are empty. The `ds-project scope` skill is intended to create PRD records, but no scope invocation has produced any prd_documents rows.

15. **F15 — PI analysis tables are empty.** All pi_* tables (analysis_runs, bugs, components, dependencies, improvements, violations, waves, wave_tasks) are empty. The `ds-analyze` skill and the 109 KB project_intelligence API router operate on empty data.

16. **F16 — Career tables are permanently empty in this context.** The 15+ career tables exist via migration 044 (the largest migration) but the `ds-career` pack explicitly requires an external system not present in this repo. These tables consume schema space and will never be populated in this deployment.

17. **F17 — workflow_invocations stores bloated JSON blobs.** The metadata_json field in workflow_invocations contains the complete node execution tree as a single JSON blob. The dedicated `execution_nodes` and `execution_outputs` tables (which have proper relational structure) are empty. The blob approach makes the node data non-queryable.

18. **F18 — research_evidence_records contains only test data.** All 9 rows reference pytest temp directories. No live research has been stored. The `research_cache` table is also empty despite the discovery research API router being wired.

19. **F19 — reg_projects and ds_projects are orthogonal but have no reconciliation.** reg_projects uses path-based slugs as project_id; ds_projects uses UUIDs. No foreign key or join logic exists between them. The telemetry system (session harvester → reg_projects) and SDLC system (project register → ds_projects) operate on separate project identity models.

20. **F20 — adapter_authority_profiles has never been written.** The adapter that is actively in use (Claude Code) has no record in adapter_authority_profiles. The schema enforces `owns_source_of_truth = 0` for all adapters, but even the read/projection metadata does not exist.

## Intent Divergence

### Intent #1 — SQLite-first authority

**Status: PARTIAL VIOLATION.**

SQLite is used for SDLC state (projects, work orders, milestones, tasks, design briefs) and the event spine. However, at least 5 distinct runtime state stores are file-backed with no SQLite equivalent: live workflow state (workflows.json), active task pointer (active_task.json), hook timing telemetry (hook-timing.jsonl), skill telemetry (skill-usage.jsonl + telemetry-buffer.jsonl), and token capture diagnostics (token-capture.jsonl). Together these represent the most frequently written runtime state in the system (the hook-timing file is 387 KB and grew during this audit session).

### Intent #2 — Security audit during brownfield onboarding

**Status: NOT MET.**

The security tables are entirely empty. The studio-onboard workflow completed successfully on 2026-05-18, but no security scan results were written to sec_* or security_findings tables. The onboarding workflow produced a text report file (`~/.dream-studio/planning/onboard-report-2026-05-18.md`) but no structured security findings in SQLite. There is no write path from any onboarding workflow node to the security tables.

### Intent #3 — Security audit as SDLC lifecycle gate

**Status: NOT MET.**

The `security_scan` post-build gate appears in multiple work order types (api_endpoint, authentication, saas_feature, data_pipeline, deployment, infrastructure). The gate mechanism checks for a filesystem artifact (`security-scan.md` in `.planning/`), not for rows in `security_findings` or `sec_*` tables. The gate is file-based (checking `.planning/security-scan.md` existence), not DB-backed. This means the gate can be satisfied by any file write, independent of whether a real security scan occurred or produced structured findings.

### Intent #4 — Canonical events as the spine

**Status: PARTIAL VIOLATION.**

The primary SDLC operations (work order start/close/block, task complete, project create/delete, milestone close) do emit canonical events via spool. However, several important mutations bypass the spine: design brief lifecycle (all operations), project state transitions (set-active, deactivate), memory/knowledge base writes (reg_gotchas, ds_documents), and the hook/tool invocation tables themselves (written directly, not via spool). The 57 validation_failures also reveal that some events that do go through spool fail schema validation and are never committed to canonical_events.

### Intent #5 — Marker file authority for attribution

**Status: ARCHITECTURE PRESENT, IMPLEMENTATION AMBIGUOUS.**

The `.dream-studio-project` marker file is read by `ds work-order start`, `ds skill invoke`, and `ds project start` for project identity resolution. This is functioning as designed. However, `reg_projects` stores project paths as a third identity source (alongside markers and ds_projects), and the relationship between reg_projects path-slug identity and ds_projects UUID identity is unresolved. The marker → ds_projects chain works; the marker → reg_projects chain does not exist.

## Open Questions

1. **Why did hook_executions stop after 2026-05-19?** The on_pulse hook wrote 45 records between 2026-05-17 and 2026-05-19, then stopped. Did the pulse hook fail silently? Was it disabled? The absence of new hook_executions rows also means the validation_failures have not accumulated further — but is this because the hook was fixed or because it stopped running?

2. **Are workflow state files intended to be permanent or transitional?** The presence of `raw_workflow_runs` alongside `workflows.json` suggests a migration from file-based to DB-based workflow state was started but not completed. Is `workflows.json` still the canonical live state, or is it a fallback that should be deprecated in favor of a DB table?

3. **What populates sec_cve_matches and sec_manual_reviews?** There is no documented write path for these tables in the Phase 0c CLI/API mapping. Were these tables created speculatively (for future tooling) or is there an undocumented hook/script that was intended to write to them?

4. **Is the 23-row test fixture pollution in ds_projects known?** The repeated `My Project` / `Programmatic Project` / `API Project` entries from 2026-05-22 test runs suggest tests are hitting the live DB. Is there a test DB isolation mechanism that has been disabled or misconfigured?

5. **What is the intended relationship between hook_invocations and canonical_events hook.tool_activity events?** With 917 rows in hook_invocations and only 246 `hook.tool_activity` events in canonical_events, what determines which tool uses generate canonical events vs DB-direct writes?

6. **Has reg_skills ever been populated?** If hydrate_registry.py was run at onboarding time and found nothing to write, that would explain the empty table. If it was never run, the registry is simply uninitialized. The 2026-05-18 onboarding workflow output mentions "9 canonical skills" were found but does not mention writing to reg_skills.

7. **Is the active project ("Dream Studio" in DB vs "Dream Command" in MEMORY.md) intentional?** The MEMORY.md was written by an AI memory system that may reflect the operator's intent, while the DB reflects the last explicit `project set-active` command. Which is authoritative, and when was the last time `project set-active` was called?

8. **Is the dual-write to workflow_invocations + raw_workflow_runs expected to continue indefinitely?** Or is the intent to eventually write to execution_nodes/execution_outputs and deprecate the blob-in-JSON approach?

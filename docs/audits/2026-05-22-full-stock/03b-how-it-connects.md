# Dream Studio — How It Connects
*Phase 3 Synthesis | 2026-05-22*

> Architecture as actually built, assessed against the v2 target model described in `02-connection-analysis.md`. The v2 target architecture document (`docs/architecture/data-model-v2.md`) does not exist. The model below is reconstructed from Phase 2 cross-reference analysis and `docs/ARCHITECTURE.md`.

---

## v2 Target Architecture (The Standard)

The v2 target specifies a 4-layer data model where all write authority flows through the canonical event spine before being projected into hub tables:

| Layer | Identity | Population rule |
|-------|----------|----------------|
| L1 Raw | Per-adapter input tables (`adapter_authority_profiles`, etc.) | Adapters write directly as their raw input stage |
| L2 Canonical | `canonical_events` only | The ONLY place writers write; all domain state changes express as events here |
| L3 Hub-and-Spoke | Business domain tables (SDLC, security, telemetry, etc.) | Populated exclusively by projections reading L2 — never by direct INSERT from code |
| L4 Views/Projections | SQL views or FastAPI derived surfaces | Derived from L3 hub tables; never written directly |

**The v2 population model:** writer emits CanonicalEventEnvelope → spool file → ingest_pending() → `canonical_events` (L2) → projection fires → L3 hub table populated → L4 view derives.

**Current state summary against this model:** L1 is entirely absent. L2 is partially correct (SDLC + token telemetry domains only). L3 is mixed — SDLC tables are paired with events but written directly, not by projection; telemetry hub tables are direct-write; 11 other domains are 0-row schema scaffolding. L4 is implemented as FastAPI routes rather than SQL views, creating a derived surface that reads from largely empty L3 tables.

## Layer-by-Layer Analysis

### L1 — Raw Adapter Input Tables

**State: Gap (does not exist in practice)**

The adapter L1 tables (`adapter_authority_profiles`, `adapter_executions`, `adapter_result_records`) were created in migration era 2 (migrations 012–034). All have 0 rows. The only implemented adapter, `claude_code`, bypasses L1 entirely and writes to L2 (spool) directly. The `last_health_state` field in `~/.dream-studio/integrations/claude_code/manifest.json` is always `null` — the health event emission guard in the adapter never fires, so the `integration.health.changed` event is never emitted, and `adapter_authority_profiles` receives no rows.

The L1 layer is a schema intent with zero operational counterpart. No path from any adapter action reaches any L1 table.

### L2 — Canonical Events

**State: Aligned for SDLC + token telemetry; Gap for everything else**

`canonical_events` has 1,853 rows. The event type distribution reveals the true scope of L2 coverage:

| Event namespace | Example types | Source | Count (approx) |
|----------------|---------------|--------|----------------|
| `tool.execution.completed` | tool use | `emitters/claude_code/run.py` → spool | 822 |
| `prompt.lifecycle.*` | prompt submission | `emitters/claude_code/run.py` → spool | 250 |
| `token.*` | `token.consumed`, `token.consumption.recorded` | `core/telemetry/token_capture.py` → spool | 150+ |
| `work_order.*` | started, closed, blocked, task.completed | `core/work_orders/mutations.py` → spool | — |
| `milestone.*` | milestone.completed | `core/milestones/close.py` → spool | — |
| `project.*` | project.registered | `core/projects/mutations.py` → spool | — |
| `context.threshold.crossed` | backfill from PostCompact normalization in run.py | Not actual threshold crossings | 20 |
| `backfill-activity-log-*` | migration-generated retroactive records | TA-series migrations (051–064) | Significant fraction |

**What is NOT in L2:**
- Session lifecycle (`session.started`, `session.ended`) — `raw_sessions` is a direct INSERT
- Handoff lifecycle — `raw_handoffs` is a direct INSERT
- Workflow lifecycle (`workflow.started`, `workflow.node.completed`) — only backfill records exist; live runs write no events
- Hook execution — `hook_invocations` / `tool_invocations` are direct INSERTs (917 rows each)
- Design brief lifecycle — all design brief mutations write L3 directly with no L2 record
- Security namespace — 0 security events; `canonical_events` has never received a `security.*` or `hook_finding.*` event
- Skill telemetry for 8 of 11 packs — `skill_invocations` has 1 row with `skill_id=unknown`
- Context threshold crossings — the import from `spool.emitter` silently fails; the 20 `context.threshold.crossed` rows are PostCompact normalization artifacts

**The backfill fraction:** A significant portion of `canonical_events` rows carry `attribution_status: "backfill"` — generated by the TA-series migrations retroactively, not by live event emission. The SDLC spine events for projects, milestones, and work orders that predated TA0 were created retroactively. This means L2's apparent completeness for SDLC is partially historical reconstruction, not forward-flowing event emission.

### L3 — Hub-and-Spoke Tables

**State: Mixed (SDLC: Drift; telemetry: Drift; all other domains: Gap)**

#### L3 SDLC Tables — Drift

`ds_projects`, `ds_work_orders`, `ds_tasks`, `ds_milestones` are the most populated and most used L3 tables. They are written by CLI mutation functions (`core/projects/mutations.py`, `core/work_orders/mutations.py`, `core/milestones/close.py`) via direct INSERT. These mutations also emit `CanonicalEventEnvelope` events through the spool. The INSERT and event emission are paired at the same call site.

This is a **Drift** from the v2 model because:
- The v2 model requires L3 to be populated by a **projection from L2**, not by direct INSERT from the mutation function.
- In practice, both happen simultaneously: the L3 row is written and the L2 event is emitted, but they are separate operations in the same function, not an event → projection chain.
- The direct INSERT is authoritative in practice; the L2 event is emitted for observability but does not drive L3 state.

The exception is `execution_events` (929 rows): this table is populated by `projections/core/execution_events_projection.py`, which is called by the spool ingestor immediately after writing to `canonical_events`. This is the **only true v2-compliant L2→projection→L3 path** in the system.

#### L3 Telemetry Tables — Drift

`hook_invocations` (917 rows) and `tool_invocations` (917 rows) — the most populated non-spool L3 tables — are written by `core/telemetry/emitters.py` via direct INSERT triggered by `on-tool-activity`. There is no corresponding L2 record. These 1,834 rows represent the largest data volume Drift in the system.

`raw_sessions` (51 rows) — direct INSERT from `on-session-start`/`on-session-end`. No L2 counterpart.

`raw_handoffs` (26 rows) — direct INSERT from `on-stop-handoff`. No L2 counterpart.

`raw_workflow_runs` (2 rows) and `workflow_invocations` (2 rows) — both written at workflow completion via direct INSERT, not via projection from any workflow-lifecycle L2 event. The two tables are parallel records for the same 2 studio-onboard runs with overlapping but non-identical schema — an unintentional dual-store.

`hook_executions` (45 rows) — direct INSERT from `on_pulse` only. Not a general hook execution record; the table name implies broader coverage that does not exist.

#### L3 Domain Tables — Gap (11 domains, all 0 rows)

Security (12 tables), PRD (9 tables), career (14 tables — external dependency), adapters (4 tables), project intelligence (8 tables), production readiness (5 tables), GitHub repo analysis (8 tables), registry (`reg_skills`, `reg_workflows`), workflow execution graph (`execution_nodes`, `execution_outputs`), billing/usage (4 tables), monitoring/alerting (4 tables).

These represent the architectural intent for domain coverage that has not been implemented. Each domain has schema (migrations created the tables) but no projection was written to populate them. The migration cadence created complete schema infrastructure ahead of any write-path work — a pattern repeated across every non-SDLC domain.

### L4 — Projections and Views

**State: Gap (implemented as FastAPI routes, not SQL views)**

The v2 specification describes L4 as SQL views derived from L3 hub tables. In practice:
- The FastAPI server at `projections/api/` has 141 routes organized into 21 route modules.
- Routes query `canonical_events`, L3 tables, and file-based stores directly.
- No SQL views over L3 exist as standalone database objects (aside from `vw_security_summary`, which unions 4 security tables that are all 0 rows).
- The L4 surface is a read API, not a derived view layer in the database.

Because most L3 domain tables have 0 rows, the L4 routes return empty data for most domains. The security, PRD, career, project intelligence, and production readiness API surfaces all return empty results. The SDLC routes return meaningful data because the L3 SDLC tables are populated.

## Major Operational Data Flows

### The SDLC Chain (Project → Milestone → Work Order → Task)

**The most complete and v2-nearest chain in the system.**

```
Operator: ds project register <name>
  → core/projects/mutations.py
    → INSERT INTO ds_projects
    → write .dream-studio-project marker file (Intent #5 authority)
    → CanonicalEventEnvelope(project.registered) → spool → canonical_events
    → execution_events_projection.apply() → INSERT INTO execution_events

Operator: ds work-order start <id>
  → core/work_orders/start.py
    → INSERT/UPDATE ds_work_orders (status=in_progress)
    → CanonicalEventEnvelope(work_order.started) → spool → canonical_events
    → execution_events_projection.apply() → INSERT INTO execution_events

Operator: ds work-order task-done <wo_id> <task_id>
  → core/work_orders/mutations.py
    → UPDATE ds_tasks (status=complete)
    → CanonicalEventEnvelope(task.completed) → spool → canonical_events

Operator: ds work-order close <id>
  → core/work_orders/close.py
    → gate checks (file-backed: security-scan.md, design_brief_locked, api_contract_exists)
    → UPDATE ds_work_orders (status=complete)
    → CanonicalEventEnvelope(work_order.closed) → spool → canonical_events
```

**Drift in this chain:**
- `ds_design_briefs` has no canonical events. `design-brief lock` (which gates WO3 from starting) writes L3 directly with no L2 record.
- `ds project set-active` and `ds project deactivate` are silent state mutations — no event.
- The `security_scan` gate resolves via `.planning/work-orders/<id>/security-scan.md` presence, not via any SQLite table.
- 23 of 25 `ds_projects` rows are test fixture contamination. The `status='active' ORDER BY updated_at DESC` resolver is theoretically vulnerable to returning a test fixture as active.

### The AI Execution Chain (Hook → Spool → Canonical Events)

**The spool pipeline: the only correctly implemented L2→L3 path.**

```
Claude Code fires hook (PostToolUse or Stop)
  → emitters/claude_code/run.py executes as subprocess
    → emitters/claude_code/emitter.py calls write_envelopes()
      → emitters/shared/spool_writer.py writes JSON files to ~/.dream-studio/events/spool/

On Stop hook:
  → spool/ingestor.py:ingest_pending() called
    → reads each pending .json file
    → validates CanonicalEventEnvelope schema
    → INSERT INTO canonical_events (L2)
    → execution_events_projection.apply() → INSERT INTO execution_events (L3)
    → moves file to events/processed/
```

**The one correct chain.** Well-tested in `tests/integration/spool/`. The model all other domains should follow.

**Spool file accumulation:** `events/processed/` has 1,593 files with no pruning mechanism. `events/.sessions/{pid}.json` has 1,102 coordination files also with no pruning. Both accumulate indefinitely.

### Token Attribution Chain

**The second most complete chain, added by TA-series workstream.**

```
Claude Code PostToolUse fires
  → core/telemetry/token_capture.py:handle_post_tool_use()
    → reads active_task.json (file-based pointer)
    → reads ds_tasks/ds_work_orders/ds_milestones/ds_projects via SQLite queries
    → if fully attributed: CanonicalEventEnvelope(token.consumed, attribution_status="fully_attributed")
    → if not attributed: CanonicalEventEnvelope(token.consumed, attribution_status="unattributed")
    → → spool → canonical_events
    → writes timing to diagnostics/token-capture.jsonl (file-based, separate path)
```

**Known gap:** `active_task.json` is the attribution pointer. If the file doesn't exist (it was absent at audit time), attribution always falls back to `unattributed`. The attribution chain depends on a file-based pointer that may be stale or absent.

**Known gap:** Token counts (`input_tokens`, `output_tokens`) are not reliably populated. `raw_token_usage` has 3 rows with all-zero values. `token_usage_records` has 0 rows. Actual token count data lands in neither SQLite table.

### The Skill Invocation Chain (Mostly Broken)

**What should happen when a skill is invoked:**
```
User: "plan: build the feature"
  → Skill tool invoked: Skill(skill="ds-core", args="plan")
  → canonical/skills/core/modes/plan/SKILL.md loads
  → skill executes
  → SHOULD: CanonicalEventEnvelope(skill.invoked, skill_id="ds-core", mode="plan", work_order_id=<current>)
    → spool → canonical_events
    → skill_invocations L3 table populated by projection
  → ACTUALLY: one of:
    a) skill.invoked emitted with work_order_id=null (ds-core:plan, 19 events — no SDLC linkage)
    b) no skill.invoked event at all (8 of 11 packs)
    c) single row in skill_invocations with skill_id=unknown
```

**Root cause of (b):** Mode-level `SKILL.md` files are not installed (only ds-bootstrap is installed per C8-L2 xfail). The skill invocation tracking hook (`on-skill-metrics.py`) writes to `skill-usage.jsonl` — but all records have `skill=unknown` because skill name resolution is failing at the hook write site.

**Root cause of (c):** `skill_invocations` was populated by the reconciliation pipeline (`core/upgrade/canonical_event_reconciliation.py`) during the TA-series, not by a live projection. The table has 1 row because the upgrade tool ran once.

## Dual-Store and Triple-Store Fragmentation

Three domains have fragmented authority — the same semantic data living in multiple places with no single authoritative source:

### Skill Telemetry — Triple Store
1. `skill-usage.jsonl` (3 lines, all `skill=unknown`) — written by `on-skill-metrics` at invocation time
2. `telemetry-buffer.jsonl` (1 line, `skill_name=unknown`) — written by `on-skill-telemetry` at Stop time
3. `raw_skill_telemetry` (0 rows) — intended destination after pulse_collector flush

None is authoritative. The pipeline is broken at stage 3. Even if fixed, stages 1 and 2 have corrupt data (`skill=unknown` in all records).

### Workflow State — Triple Store
1. `workflows.json` — live state during workflow execution (currently empty; `{}`)
2. `raw_workflow_runs` (2 rows) — archived at completion via direct INSERT
3. `workflow_invocations` (2 rows) — parallel table, same 2 runs, slightly different schema

The `raw_*` tables appear to be v1 persistence that was not replaced when `workflow_invocations` was introduced. Execution graph tables (`execution_nodes`, `execution_outputs`) exist in schema but have 0 rows despite workflow runs having occurred.

### Hook Execution — Triple Store
1. `hook-timing.jsonl` (3,645 lines, 392 KB) — written by `dispatch_helpers.write_timing()` for every hook dispatch
2. `hook_executions` (45 rows) — written by `on_pulse` ONLY; not a general hook execution record
3. `hook_invocations` (917 rows) — written by `on-tool-activity` ONLY for tool-use events

No single table captures all hook executions. `hook-timing.jsonl` is the only record with full coverage. The analytics harvester reads the full file on every `ds analytics` invocation to compute per-handler averages — a full file scan where a SQL GROUP BY query would be more efficient and correct.

### Projects — Dual-Table with Identity Mismatch
1. `reg_projects` (2 rows) — `project_id` is a directory name (e.g., `dream-studio-clean`); written by `on-session-start` using `Path.cwd().name`
2. `ds_projects` (25 rows) — `project_id` is a UUID; written by `ds project register`

These are distinct registries for different consumers (analytics vs SDLC). They use incompatible identity schemes and cannot be joined without a bridge table. The confusion is compounded by both being called "projects" in their table names.

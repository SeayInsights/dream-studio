# Migration Sequence

Migrations are numbered `NNN_description.sql` and applied in ascending order by the runner in `core/config/sqlite_bootstrap.py`. Applied versions are tracked in the `_schema_version` table.

## 2026-07-04 — Baseline squash (WO-SQUASH-BASELINE, 5fd84891)

Operator-approved irreversible squash: migrations 001-141 were collapsed into
a single lean baseline, `142_lean_baseline.sql`. The 139 individual migration
files that made up that chain (001-141, minus the two intentional gaps below)
were deleted from the working tree in the same change set — they remain in
git history for archaeology but are no longer part of this directory or the
runner's glob. `142_lean_baseline.sql` was generated by applying the full
pre-squash chain to a fresh DB and re-emitting every resulting schema object
(tables, indexes, views, triggers) in idempotent `CREATE ... IF NOT EXISTS`
form, preceded by `DROP TABLE IF EXISTS` / `DROP VIEW IF EXISTS` for every
tombstoned name (see `tests/unit/schema_tombstones_data.py`). See
`docs/MIGRATION_AUTHORITY.md` for the full pre-squash migration-by-migration
history and `docs/architecture/aspirational-schema-debt.md` for the dated
review entry. New migrations start at **143**.

## Adding new migrations

1. Use the next sequential number (**143** as of the 2026-07-04 squash).
2. Name the file `NNN_short_description.sql`.
3. Every statement must end with `;`.
4. Prefer `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` for idempotent DDL.
5. `ALTER TABLE ... ADD COLUMN` is safe; SQLite does not support `IF NOT EXISTS` for it, but the runner skips `duplicate column name` errors.
6. Run the full migration regression test before pushing: `py -m pytest tests/integration/migrations/test_full_migration_sequence.py -v`
